Yes. AF-UI gives us a very useful UI-side analogue to the backend/domain primitives we’ve been designing.

The strongest lesson is: **generated UI should not be “React components with props and class strings.”** It should be a typed, platform-agnostic UI IR with structure, slots, styles, behaviors, requirements, and render targets separated. AF-UI’s key split is `Tags → Views → Components → Renderer → Platform`, and components return a typed `View<Slots>` rather than a black-box `JSX.Element`. That is exactly the kind of primitive our `@gen/forms` / `@gen/components` layer should learn from. 

## What to take

### 1. Typed slots / handles should be first-class

AF-UI’s best idea is that a component exposes named slots like `root`, `label`, `content`, `closeButton`, etc., and styles/behaviors attach to those slots from the outside. That fits perfectly with generated forms and design-system customization. 

In our spec, instead of generating this as opaque JSX:

```tsx
function UserForm() {
  return <form className="...">...</form>
}
```

we should generate a typed view:

```ts
const UserCreateView = gen.ui.view({
  root: gen.ui.Element.Form,
  emailField: gen.ui.Element.Field,
  emailInput: gen.ui.Element.TextInput,
  submitButton: gen.ui.Element.Interactive,
}, ({ slots }) => (
  <Form slot={slots.root}>
    <Field slot={slots.emailField}>
      <TextInput slot={slots.emailInput} />
    </Field>
    <Button slot={slots.submitButton}>Create</Button>
  </Form>
))
```

Then external packages can attach:

```ts
const StyledUserCreateForm = UserCreateView.pipe(
  gen.ui.Style.attach({
    root: { gap: "md" },
    emailInput: { borderColor: "neutral" },
    submitButton: { backgroundColor: "primary" },
  }),
  gen.ui.Behavior.attach(submitOnEnter, {
    target: "root",
  })
)
```

That means users can customize generated components without forking generated files.

This solves the same “shadcn fork rot” problem AF-UI describes: generated code should expose typed handles, not require copy-paste edits. 

---

### 2. Separate View, Style, Behavior, and Platform

Our current plan has forms/components, but AF-UI gives us a better breakdown:

```txt
View       = structure + slots
Style      = appearance attached to slots
Behavior   = interaction attached to slots
Component  = logical state + view
Renderer   = compiles/interprets UI IR
Platform   = web/native/TUI vocabulary + events + rendering
```

This maps well to our static-function design:

```txt
Form Func       = typed action/input contract
Form View       = structural UI over that input
Form Behavior   = submit, validation, focus, disclosure, keyboard nav
Form Style      = tokens, variants, responsive layout
Renderer        = React/Vue/Svelte/Native/TUI target
Platform        = Web DOM, React Native, terminal, server components, etc.
```

So generated forms should be:

```ts
const UserCreateForm = gen.forms.fromFunc(createUserFunc, {
  view: gen.ui.view(...),
  behavior: gen.forms.behavior.standardSubmit(),
  style: gen.forms.style.defaultForm(),
})
```

rather than directly “generate React component.”

---

### 3. Behaviors should attach to capabilities, not component names

AF-UI’s behavior system matches behaviors to element capabilities like `Element.Interactive`, `Element.TextInput`, `Element.Container`. That is excellent. 

For us:

```ts
const submitBehavior = gen.ui.Behavior.make({
  form: gen.ui.Element.Form,
  submit: gen.ui.Element.Interactive,
})(({ form, submit }) =>
  gen.action.seq(
    submit.onPress(form.submit()),
    form.onSubmit(createUserFunc)
  )
)
```

Then a behavior can only attach to compatible slots:

```ts
UserCreateView.pipe(
  gen.ui.Behavior.attach(submitBehavior, {
    form: "root",
    submit: "submitButton",
  })
)
```

This should fail:

```ts
gen.ui.Behavior.attach(submitBehavior, {
  submit: "emailInput",
})
```

if `emailInput` is `TextInput` but not `Interactive`, or does not satisfy the required capability.

This is exactly analogous to our typed relation/field refs: **UI slots are refs with capabilities**.

---

### 4. Typed design tokens beat Tailwind strings

AF-UI’s critique of Tailwind strings is directly relevant. Generated UI should not emit giant untyped class strings as the source of truth. Styles should be typed data using tokens. 

We should add a style/token primitive:

```ts
const AppTheme = gen.ui.theme({
  color: {
    primary: gen.ui.color("#2563eb"),
    surface: gen.ui.color("#ffffff"),
    danger: gen.ui.color("#dc2626"),
  },
  space: {
    xs: "0.25rem",
    sm: "0.5rem",
    md: "1rem",
  },
  radius: {
    sm: "0.25rem",
    md: "0.5rem",
    full: "9999px",
  },
})
```

Then styles:

```ts
const buttonStyle = gen.ui.Style.make({
  root: {
    padding: ["sm", "md"],
    borderRadius: "md",
    backgroundColor: "primary",
  },
  label: {
    color: "surface",
    fontWeight: "bold",
  },
})
```

Invalid token names should be type errors.

Targets can compile this to:

```txt
CSS classes
Tailwind-compatible classes
inline styles
React Native StyleSheet
terminal styles
static extracted CSS
```

but the source remains typed.

### 5. Generated UI should have style handles

This is a big one.

For every generated form/table/detail/card, expose public handles:

```ts
const UserCreateForm = gen.forms.fromFunc(createUserFunc, {
  handles: {
    root: gen.ui.Element.Form,
    emailField: gen.ui.Element.Field,
    emailInput: gen.ui.Element.TextInput,
    displayNameField: gen.ui.Element.Field,
    displayNameInput: gen.ui.Element.TextInput,
    submitButton: gen.ui.Element.Interactive,
  },
})
```

Then users or design-system plugins can override safely:

```ts
const BrandedUserCreateForm = UserCreateForm.pipe(
  gen.ui.Style.attach({
    submitButton: {
      backgroundColor: "primary",
      borderRadius: "full",
    },
  })
)
```

No generated-file fork. No magic CSS selectors. No “find the right div.”

---

### 6. Requirements should bubble through UI too

AF-UI uses Effect-style requirements: if a child component needs `Auth` or `Router`, the parent’s required services include that automatically. 

This lines up with our `Runtime`, `Effect`, and `Requirement` model.

A generated form might require:

```txt
Router
MutationClient
ToastService
Theme
AuthContext
I18n
```

So the UI function/component should carry requirements:

```ts
const UserCreateComponent = gen.ui.component({
  name: "UserCreateComponent",
  props: gen.object({}),
  requires: [
    gen.req.service("MutationClient"),
    gen.req.service("Theme"),
    gen.req.service("Router"),
  ],
  view: ...
})
```

When composed into a page/layout, requirements bubble upward. The generator can fail if the target app shell does not provide them.

---

### 7. Views should support slot remapping

AF-UI’s slot remapping example is very relevant for generated UI composition. 

For example, a generated `ModalForm` might wrap `UserCreateForm` but remap slots:

```ts
const UserCreateModal = gen.ui.view({
  modalRoot: gen.ui.Element.Container,
  modalTitle: gen.ui.Element.Text,
  modalBody: gen.ui.Element.Container,
  formSubmit: gen.ui.Element.Interactive,
}, () => (
  <Modal
    slots={{
      root: "modalRoot",
      title: "modalTitle",
      content: "modalBody",
    }}
  >
    <UserCreateForm
      slots={{
        submitButton: "formSubmit",
      }}
    />
  </Modal>
))
```

This lets generated components compose without leaking internal structure.

---

### 8. Collection slots matter for generated lists/tables

AF-UI’s `Element.Collection<Element.Container>` is useful for generated tables, lists, relation selects, data grids, and repeatable form sections. 

We should add collection slots:

```ts
const UsersTableView = gen.ui.view({
  root: gen.ui.Element.Table,
  rows: gen.ui.Element.Collection(gen.ui.Element.Row),
  cells: gen.ui.Element.Collection(gen.ui.Element.Cell),
})
```

Then selection, keyboard navigation, drag-and-drop, row expansion, bulk actions, and virtualization can be behaviors attached to typed collections:

```ts
UsersTableView.pipe(
  gen.ui.Behavior.attach(selectionBehavior, {
    target: "rows",
  }),
  gen.ui.Behavior.attach(keyboardNavBehavior, {
    target: "rows",
  })
)
```

This is much better than generated ad hoc table components.

## The most important design lesson

AF-UI’s inside-out model solves a problem we will definitely have: **users will want to customize generated UI without editing generated files.**

So our generated UI should never be a sealed black box.

It should be:

```txt
typed function contract
+ generated structural view with slots
+ attachable behavior
+ attachable style
+ platform renderer
```

That gives us:

```txt
type-safe generated forms
type-safe component customization
no generated-file forking
no Tailwind string typos
platform-agnostic render targets
runtime/requirement bubbling
behavior reuse
design-system override hooks
```

Add a new major section after Forms or before Components:

````diff
+# 23. UI IR
+
+## 23.1 Purpose
+
+`@gen/ui` defines a platform-agnostic intermediate representation for generated UI.
+
+It should be used by:
+
+```txt
+forms
+components
+tables
+detail views
+cards
+modals
+admin pages
+design-system packages
+framework renderers
+```
+
+The purpose is to avoid generating sealed framework components that users must fork to customize.
+
+## 23.2 Architecture
+
+The UI stack is:
+
+```txt
+Element capabilities -> View<Slots> -> Component -> Renderer -> Platform
+```
+
+Definitions:
+
+```txt
+Element capability:
+  abstract element interface such as Container, Text, Interactive, TextInput, Form.
+
+View<Slots>:
+  structural skeleton with named typed slots.
+
+Component:
+  logical unit with props, requirements, possible errors, bindings/state, and a View.
+
+Style:
+  typed appearance data attached to slots.
+
+Behavior:
+  typed interaction/effect/action data attached to slots.
+
+Renderer:
+  service that converts UI IR into host nodes or source files.
+
+Platform:
+  element vocabulary, event system, renderer, attribute model, and host capabilities.
+```
+
+## 23.3 Element Capabilities
+
+The UI package should define abstract element capabilities:
+
+```ts
+gen.ui.Element.Base
+gen.ui.Element.Container
+gen.ui.Element.Text
+gen.ui.Element.Interactive
+gen.ui.Element.TextInput
+gen.ui.Element.NumberInput
+gen.ui.Element.Select
+gen.ui.Element.Form
+gen.ui.Element.Label
+gen.ui.Element.Field
+gen.ui.Element.Table
+gen.ui.Element.Row
+gen.ui.Element.Cell
+gen.ui.Element.Collection(inner)
+```
+
+Element capabilities are not hardcoded HTML tags.
+
+Platform packages map them to concrete tags/widgets/events:
+
+```txt
+Web:
+  Interactive -> button, a, div[role=button], etc.
+  onPress -> click / keyboard activation
+
+TUI:
+  Interactive -> focusable terminal node
+  onPress -> key event / enter
+
+Native:
+  Interactive -> Pressable
+  onPress -> GestureResponderEvent
+```
+
+## 23.4 View<Slots>
+
+A view is a typed structural skeleton.
+
+```ts
+const BaseButtonView = gen.ui.view({
+  root: gen.ui.Element.Interactive,
+  label: gen.ui.Element.Text,
+}, ({ props, slot }) => (
+  <Button slot={slot.root}>
+    <Text slot={slot.label}>{props.children}</Text>
+  </Button>
+))
+```
+
+The stored artifact should be a UI AST, not an opaque JSX element.
+
+The callback is an authoring surface, like `gen.expr`; it builds static UI structure.
+
+## 23.5 Component
+
+Components combine props, setup/bindings, requirements, and a view.
+
+```ts
+const BaseButton = gen.ui.component({
+  name: "BaseButton",
+  props: gen.object({
+    children: gen.ui.node(),
+  }),
+  requires: [],
+  setup: gen.func.expr({
+    name: "buttonSetup",
+    input: gen.object({}),
+    returns: gen.object({}),
+    body: gen.expr(() => ({})),
+  }),
+  view: BaseButtonView,
+})
+```
+
+Conceptual type:
+
+```ts
+type Component<Props, Req, E, Bindings, Slots> = {
+  kind: "component"
+  props: Props
+  requirements: Req
+  errors: E
+  bindings: Bindings
+  view: View<Slots>
+  pipe: PipeableComponent<Props, Req, E, Bindings, Slots>
+}
+```
+
+Components should not return black-box `JSX.Element`.
+They should return or contain a typed `View<Slots>`.
+
+## 23.6 Style
+
+Styles are typed data attached to slots.
+

+const buttonStyle = gen.ui.Style.make({
+  root: {
+    padding: ["sm", "md"],
+    borderRadius: "md",
+    backgroundColor: "primary",
+  },

+    color: "inverse",
+    fontWeight: "bold",
+  },
+})

+)
+```
+
+Invalid slot names should be type errors.
+
+Invalid theme tokens should be type errors.
+
+Invalid style properties for a platform should be diagnostics or type errors where possible.
+
+## 23.7 Theme and Typed Tokens
+
+The UI package should support typed theme tokens.

+  },

+```
+
+Styles should refer to tokens:
+
+```ts

+padding: ["sm", "md"]

+```
+

+
+Generated components should expose public style handles.

+This avoids generated-file forking.

+## 23.9 Behavior
+
+Behaviors are typed slot consumers that attach interaction, accessibility, state,
+subscriptions, lifecycle resources, or effects to slots.
+
+```ts
+const submitBehavior = gen.ui.Behavior.make({
+  form: gen.ui.Element.Form,
+  submit: gen.ui.Element.Interactive,
+})(({ form, submit }) =>
+  gen.action.seq(
+    submit.onPress(form.submit()),
+    form.onSubmit(createUserFunc)
+  )
+)
+```
+
+Attach:
+
+```ts
+const UserCreateFormWithBehavior = UserCreateForm.pipe(
+  gen.ui.Behavior.attach(submitBehavior, {
+    form: "root",
+    submit: "submitButton",
+  })
+)
+```
+
+Attachment should type-check:
+
+```txt
+slot exists
+slot satisfies required element capability
+behavior requirements are provided or bubble upward
+events are supported by target platform
+effects are supported by target runtime
+```
+
+## 23.10 Lifecycle and Cleanup
+
+Behaviors may acquire resources:
+
+```txt
+event listeners
+timers
+subscriptions
+focus traps
+observers
+external stores
+```
+
+The behavior system should model lifecycle requirements and cleanup as static action/effect plans where possible.
+
+Platform renderers are responsible for translating lifecycle plans into concrete cleanup behavior.
+
+## 23.11 Slot Remapping
+
+Views/components should support slot remapping for composition.
+
+```ts
+const Panel = gen.ui.component({
+  name: "Panel",
+  props: gen.object({
+    title: gen.string(),
+    children: gen.ui.node(),
+  }),
+  view: gen.ui.view({
+    panelRoot: gen.ui.Element.Container,
+    panelTitle: gen.ui.Element.Text,
+    panelBody: gen.ui.Element.Container,
+  }, ({ props, slot }) => (
+    <BaseModal
+      title={props.title}
+      slots={{
+        root: slot.panelRoot,
+        title: slot.panelTitle,
+        content: slot.panelBody,
+      }}
+    >
+      {props.children}
+    </BaseModal>
+  )),
+})
+```
+
+Slot remapping lets components compose without leaking internal slot names.
+
+## 23.12 Collection Slots
+
+Generated lists, tables, grids, repeatable form sections, and relation selectors
+need typed collection slots.
+
+```ts
+const UsersTableView = gen.ui.view({
+  root: gen.ui.Element.Table,
+  rows: gen.ui.Element.Collection(gen.ui.Element.Row),
+  cells: gen.ui.Element.Collection(gen.ui.Element.Cell),
+})
+```
+
+Behaviors can attach to collections:
+
+```ts
+UsersTableView.pipe(
+  gen.ui.Behavior.attach(selectionBehavior, {
+    target: "rows",
+  }),
+  gen.ui.Behavior.attach(keyboardNavBehavior, {
+    target: "rows",
+  })
+)
+```
+
+This supports:
+
+```txt
+selection
+keyboard navigation
+drag and drop
+virtualization
+row expansion
+bulk actions
+sortable tables
+```
+
+## 23.13 Requirements Bubble Through UI
+
+Components and behaviors should declare requirements:
+
+```txt
+Theme
+Router
+MutationClient
+QueryClient
+ToastService
+AuthContext
+I18n
+Platform capabilities
+```
+
+When a component renders a child or attaches a behavior with requirements,
+those requirements should bubble up to the parent component/page/app.
+
+Missing requirements should be compile-time errors where possible or structured diagnostics.
+
+## 23.14 Platform and Renderer
+
+Platform packages define:
+
+```txt
+element vocabulary
+attribute model
+event model
+renderer
+host capabilities
+style output
+lifecycle behavior
+```
+
+Examples:
+
+```ts
+gen.web.platform(...)
+gen.react.renderer(...)
+gen.vue.renderer(...)
+gen.native.platform(...)
+gen.tui.platform(...)
+```
+
+Generated UI targets should compile the same `@gen/ui` IR to platform-specific output.
+
+## 23.15 Security Holes
+
+UI holes that can introduce security risks should require branded safe types.
+
+Example:
+
+```ts
+gen.ui.html(SafeHtml)
+```
+
+not:
+
+```ts
+gen.ui.html(gen.string())
+```
+
+Unsafe HTML/string interpolation should be explicit and diagnosable.
